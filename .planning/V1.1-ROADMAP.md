# OpenArangoDBCore v1.1 Roadmap

## Overview

v1.0 delivered all 19 ArangoDB Enterprise modules (565 tests, 0 failures). v1.1 focuses on **production hardening, performance optimization, and community readiness** — the gap between "feature complete" and "production ready."

## Priorities

| Priority | Theme | Effort | Impact |
|----------|-------|--------|--------|
| P0 | Production Hardening | 2-3 weeks | Critical for adoption |
| P1 | Performance Optimization | 2-3 weeks | 30-50% gains in key paths |
| P2 | Community & Packaging | 2-3 weeks | Enables external contributors |
| P3 | Extended Testing | 2-3 weeks | Reliability confidence |

## P1: Performance Optimization

### 1.1 — AES-NI / Hardware Acceleration for Encryption
- Current: ~300 MB/s (pure OpenSSL)
- Target: ~1.2 GB/s with AES-NI / AVX2
- Enable hardware acceleration in OpenSSL configuration
- Add benchmark comparison before/after
- **Estimated gain: 3-4x encryption throughput**

### 1.2 — LDAP Connection Pooling
- Current: Per-request LDAP connection (handshake on every auth)
- Target: Connection pool with TTL-based eviction
- Add pool size and TTL configuration options
- **Estimated gain: 50% auth latency reduction**

### 1.3 — SmartGraph Hash Optimization
- Current: FNV-1a for shard routing (~18M hashes/sec)
- Evaluate: xxHash64 (10-15x faster, non-cryptographic, fine for sharding)
- Must verify hash distribution doesn't change shard assignment semantics
- **Estimated gain: 15-20% traversal overhead reduction**

### 1.4 — Audit Logging Zero-Copy
- Current: Ring buffer with string copy
- Target: Lock-free queue with zero-copy serialization
- Add JSON structured output option (currently text)
- **Estimated gain: <1% write latency improvement**

### 1.5 — Link-Time Optimization (LTO)
- Enable LTO for release builds
- Strip/debug symbol separation
- Reproducible builds verification
- **Estimated gain: 5-10% binary size reduction, 2-5% runtime improvement**

## P3: Extended Testing

### 3.1 — Edge Case Unit Tests (+80-100 tests)
**Encryption:**
- Key rotation rollback, key file corruption recovery
- Concurrent key rotation during writes

**LDAP:**
- Server timeout/unavailable scenarios
- Secondary server failover, TLS cert validation failure

**SmartGraph:**
- Shard addition/removal during graph operations
- Rebalancing during traversal
- Nested SmartGraph scenarios

**Data Masking:**
- Deeply nested path masking (`users[0].contact.email`)
- Array element masking, dynamic schema masking

**ReadFromFollower:**
- Read-after-write consistency
- Stale read bounds validation
- Leader/follower switchover during read

### 3.2 — Stress & Chaos Tests
- Hot backup during peak write load
- Cloud backup credential rotation mid-transfer
- Parallel index build with concurrent modifications
- DC2DC replication with large blobs (>100MB documents)
- Network partition simulation for replication
- Process crash recovery tests

### 3.3 — Cluster Integration Tests
- Multi-server cluster setup and teardown
- Real shard distribution verification
- Cross-shard traversal correctness
- Satellite collection replication timing

### 3.4 — Performance Regression Tests
- Benchmark CI job with threshold alerts
- Historical benchmark tracking
- Regression detection for key metrics (encryption throughput, hash rate, replication latency)

### 3.5 — Observability
- Prometheus metrics export (replication lag, backup duration, encryption overhead)
- OpenTelemetry tracing for DC2DC replication
- Structured JSON logging option
